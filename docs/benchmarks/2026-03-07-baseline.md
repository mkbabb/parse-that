### Work equivalence notes
- parse_that, BBNF-generated, nom, winnow, pest: zero-copy borrowed `&str`, raw escapes
- BBNF-generated: `#[derive(Parser)]` from `.bbnf` grammar — zero hand-written Rust
- serde_json: owned `String`, fully decoded escapes (MORE work)
- serde_json_borrow: `Cow<str>`, decoded escapes
- jiter: `Cow<str>`, selectively decoded
- simd-json: `Cow<str>` borrowed mode, `.to_vec()` per iteration (mutable input)
- sonic-rs: owned `Value`, arena alloc, fully decoded (MORE work, SIMD-accelerated)

### Summary
parse_that is **1.5-1.7x faster than nom/winnow** (work-equivalent peers). BBNF-generated parsers reach 60-70% of hand-rolled throughput with zero hand-written Rust. SIMD parsers (simd-json, sonic-rs) are 1.5-3x faster but use platform-specific SIMD instructions.

## Rust CSS (MB/s)

| Parser | normalize (6KB) | bootstrap (281KB) | tailwind (3.6MB) |
|---|---|---|---|
| **parse_that** (L1.5) | **283** | **237** | **267** |
| BBNF-generated (L1) | 241 | 67 | — |
| cssparser (L0-L1) | 686 | 434 | 272 |
| lightningcss (L2) | 227 | 104 | FAIL (parse error) |

### Notes
- parse_that CSS uses monolithic byte-level scanners (css_ident_fast, css_ws_comment_fast with memchr, css_string_fast with memchr2) and inline byte dispatch — no combinator vtable calls on hot paths
- BBNF-generated parser uses `#[derive(Parser)]` with `css-fast.bbnf` — opaque spans, no typed values/selectors (LESS work than parse_that)
- cssparser is servo's tokenizer with callback visitors (LESS work — no AST)
- lightningcss does L2 semantic parsing (MORE work — property validation, shorthand expansion)
- parse_that is **1.25x faster than lightningcss** on normalize, **2.3x faster** on bootstrap — while doing comparable L1.5 work
- parse_that is within **1.6-2.4x of cssparser** while building a full typed AST vs tokenization only
- On tailwind (3.6MB), parse_that at 267 MB/s is essentially tied with cssparser at 272 MB/s

## TypeScript CSS

| Parser | normalize (6KB) | bootstrap (274KB) |
|---|---|---|
| **parse-that** (L1.5) | **1.00x** | **1.00x** |
| postcss (L1) | 3.86x slower | 2.74x slower |
| css-tree (L1-L2) | 5.70x slower | 4.40x slower |

### Fairness validation
- Declaration counts match exactly: normalize 57/57, bootstrap 5542/5542
- parse-that produces typed values (`{type: "dimension", value: 100, unit: "px"}`) while postcss returns raw strings (`"100px"`)
- parse-that produces typed selectors (compound/complex AST) while postcss returns raw strings
- parse-that does MORE semantic work than postcss — the 2.7-3.9x advantage is conservative
