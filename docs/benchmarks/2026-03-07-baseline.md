### Work equivalence notes
- parse_that, BBNF-generated, nom, winnow, pest: zero-copy borrowed `&str`, raw escapes
- BBNF-generated: `#[derive(Parser)]` from `.bbnf` grammar — zero hand-written Rust
- serde_json: owned `String`, fully decoded escapes (MORE work)
- serde_json_borrow: `Cow<str>`, decoded escapes
- jiter: `Cow<str>`, selectively decoded
- simd-json: `Cow<str>` borrowed mode, `.to_vec()` per iteration (mutable input)
- sonic-rs: owned `Value`, arena alloc, fully decoded (MORE work, SIMD-accelerated)

### Summary
parse_that is **1.0–1.7x faster than nom/winnow** (work-equivalent peers). BBNF-generated parsers reach 60–70% of hand-rolled throughput with zero hand-written Rust. SIMD parsers (simd-json, sonic-rs) are 1.5–3x faster but use platform-specific SIMD instructions.

## Rust CSS (MB/s)

| Parser | normalize (6KB) | bootstrap (281KB) | tailwind (3.6MB) |
|---|---|---|---|
| **parse_that** (L1.75) | **521** | **265** | **259** |
| BBNF-generated (L1) | 231 | 65 | — |
| cssparser (L0-L1) | 679 | 436 | 274 |
| lightningcss (L2) | 230 | 110 | — |

### Notes
- parse_that CSS (L1.75) produces a fully typed AST: selectors (compound/complex), values (dimension/color/function), typed media queries (MediaQuery/MediaCondition/MediaFeature with range ops), typed @supports conditions, and specificity computation
- Monolithic byte-level scanners (css_ident_fast, css_ws_comment_fast with memchr, css_string_fast with memchr2) and inline byte dispatch — no combinator vtable calls on hot paths
- BBNF-generated parser uses `#[derive(Parser)]` with `css-fast.bbnf` — opaque spans, no typed values/selectors (LESS work than parse_that)
- cssparser is Servo's tokenizer with callback visitors (LESS work — no AST)
- lightningcss does L2 semantic parsing (MORE work — property validation, shorthand expansion)
- parse_that is **2.3–2.4x faster than lightningcss** on normalize and bootstrap — while doing comparable L1.75 work
- On tailwind (3.8MB), parse_that at 259 MB/s is within 1.06x of cssparser at 274 MB/s — competitive with a tokenizer-only parser while building a full typed AST

### Fairness validation
- Declaration counts match exactly: normalize 57/57, bootstrap 5542/5542
- parse-that produces typed values (`{type: "dimension", value: 100, unit: "px"}`) while postcss returns raw strings (`"100px"`)
- parse-that produces typed selectors (compound/complex AST), typed media queries, typed @supports conditions, and specificity — while postcss returns raw strings
- parse-that does MORE semantic work than postcss — the 1.6–2.3x advantage is conservative

### Notes
- gorgeous uses `#[derive(Parser)]` with a `.bbnf` grammar + `@pretty` directives -- zero hand-written formatting code
- `@pretty selectorSpan split(",") group sep(", ")` -- format-time balanced splitting replaces runtime override
- `split_balanced()` in parse_that: memchr fast-path + `()[]""''`-aware depth tracking
- `contains_delimiter()` guard in generated code eliminates Vec allocation for single-selector rules (~95% of rules)
- pprint render optimizations: `LinearJoin` variant (no text_justify pre-pass), reverse cursor SmartJoin, FxHashMap pre-allocated (256 capacity)
- gorgeous is **2.5--19x faster than biome** depending on file size
